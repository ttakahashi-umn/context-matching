# Brief: talent-interview-profile-poc

## Problem

- 人材の「文脈」は、職務経歴書や提携向けの定型情報だけでは相手に伝わりにくい。
- 非定型の情報（面談で語られる経験・価値観・暗黙知に近いもの）は、**入力が重い**一方で、マッチング品質に効く可能性が高い。
- まずは **人材側のデータ基盤** から検証し、事業側（運営）視点で「筋が良い」と判断できる材料を作りたい。

## Current State

- **API（FastAPI）・最小 Web UI（Vite + React）・SQLite** を含む PoC アプリケーションがリポジトリに実装されている。詳細な契約と境界は **`.kiro/specs/talent-interview-profile-poc/`**（要件・設計・タスク）および steering（product / tech / structure）に記載する。
- PoC は **近親者レベル** の利用想定で、本番の金銭・契約は扱わない。

## Desired Outcome

- **営業の役員**が、提示物を見て **「筋がいい」** と言える状態。
- 具体的には、**面談のボイス→テキスト** を起点に、**YAML テンプレで定義した観点** に沿ってテキストを解析し、**人材プロフィールとして保存できる構造化データ** が得られること（デモで再現可能であること）。
- 発注側プロフィール、マッチング、**本番運用を前提としたフルスペックの運営コンソール**、同意・削除・監査の本格実装は **この PoC では扱わない**（**最小 PoC UI** は In）。

## Approach

- **スコープを人材データベースに限定**する。発注側（会社・部門・案件）のプロフィールは **今回やらない**。
- **取り込みソース**は当面 **インタビュー文字起こし（ボイス→テキスト済みテキスト）** に限定する。SNS・GitHub 等は将来拡張として扱い、本 brief の In には含めない。
- **抽出の設計**は、取り込みたい観点・キーワード群を **YAML 等のテンプレ** で宣言し、そのテンプレに従って **LLM で解析・構造化**する。
- **推論環境**は、可能なら **Mac 上のオンデバイス推論（CoreML を含む Apple 系スタック）** を優先する。採用する具体モデル・ランタイムは設計フェーズで確定し、PoC 段階で実行可能性を検証する。
- **本番運用を前提としたフルスペックの運営コンソールは作らない**（従来の D に相当する広い運営 UI は外）。一方で **最小の PoC UI**（コア業務を画面から辿れること）は **In** とする。役員への提示は、最小 PoC UI に加え、**構造化データのエクスポート、再現可能な実行、読みやすい静的サマリ** などを併用できる（形式は設計で具体化する）。

## Scope

- **In**
  - 人材側エンティティの **保存・更新・参照**（最低限のデータモデル）。
  - 面談テキスト（文字起こし済み）の **取り込みパス**（ファイル投入や貼り付け等、実装は設計で決定）。
  - **YAML（等）テンプレ** による抽出観点の宣言とバージョン管理。
  - テンプレに基づく **LLM による解析パイプライン**（オンデバイス優先）。
  - PoC 用の **再現手順**（同じ入力で同様の構造化結果が得られることの説明可能な範囲）。
  - **最小 PoC UI**（人材・面談・テンプレ・抽出・プロフィールの主要操作を画面から実行・閲覧できること）。
- **Out（明示的に含めない）**
  - 発注側・案件側のプロフィール（A の発注側）。
  - マッチング、ランキング、マッチ理由の自動生成（C）。
  - **本番運用を前提としたフルスペックの運営コンソール**およびそれに相当する広いワークフロー（従来の D のうち本番運営寄りの部分）。
  - 同意管理・削除権・監査ログ等の本番ガバナンス（E）。※近親者 PoC でも、後続で詰めるべき論点として design にメモする余地はある。
  - SNS・プレス・GitHub 等の **追加ソースコネクタ**（今回の In 外。将来の隣接仕様）。

## Boundary Candidates

- **「人材レコード」 vs 「面談セッション」**: 1 人に複数面談がある前提で、生テキストと抽出結果の履歴をどう紐づけるか。
- **「テンプレ（YAML）」 vs 「プロフィールスキーマ」**: テンプレ変更が既存プロフィールに与える影響（再解析の要否）。
- **「オンデバイス LLM」 vs 「将来のクラウド」**: PoC はオンデバイス優先だが、フォールバック方針は設計で明示するか、完全に禁止するか。

## Out of Boundary

- 収益モデル、契約、課金。
- 本番レベルのセキュリティ運用（ただしデータの扱いの最低ラインは steering と整合させる）。

## Upstream / Downstream

- **Upstream**: ボイス→テキストの品質（外部ツール／別プロセスと仮定してよい）。面談の取得プロセス自体は本仕様の主対象外でもよいが、入力形式（プレーンテキスト等）は固定する。
- **Downstream（将来）**: 発注側プロフィール、マッチング、運営 UI、追加データソース、ガバナンス。いずれも別 spec または本 spec のフェーズ 2 以降。

## Existing Spec Touchpoints

- **Extends**: なし（初回 spec）。
- **Adjacent**: steering の product.md（全体ビジョン）と整合。将来のマッチング仕様と重複しないよう、本 PoC は **人材プロフィール構築** に閉じる。

## Constraints

- 実行環境は **Mac を主** とし、**CoreML を含むオンデバイス推論** を嬉しい要件として扱う。
- 利用者は **近親者レベル** の PoC。個人情報は扱うが、本番相当の法令対応までは求めない（E 外）。それでも **不要なデータの保持を避ける** など、最低限の配慮は steering に従う。
