# Brief: project-registration-poc

## Problem

マッチングサービスとしては **案件（募集・オーダ・求めている役割と条件）** がデータとして存在することが前提だが、現リポジトリの実装済み PoC（`talent-interview-profile-poc`）は **人材と面談プロフィール** に限定しており、案件側の当事者データを登録・参照する手段がない。そのため、プロダクト全体の文脈に沿った **最小の案件マスタ** を別境界として追加する必要がある。

## Current State

- `apps/api` / `apps/web` に人材・面談・テンプレ・抽出・プロフィール反映の縦割りが実装済み。
- `talent-interview-profile-poc` の設計上、**発注側・案件エンティティは対象外**として明示されている。
- 案件に関する REST・UI・永続化モデルは **未実装**。

## Desired Outcome

運用者（または将来の発注側ユーザー）が、**案件を登録し、一覧・詳細で再参照し、必要なら内容を更新できる**。登録内容には少なくとも **識別可能なタイトルと説明（または要件本文）** が含まれ、システムは **一意な識別子** を返す。人材プロフィール PoC と **データ上は独立**（同一 DB ファイルに載せるかは設計で決めるが、ドメイン境界は分離する）。

## Approach

既存モノレポ（Python API + React Web + SQLite）のパターンに揃え、**新しい機能境界**として「案件」エンティティを追加する。レイヤード構成・REST・OpenAPI 露出・最小 Web UI（または API のみの先行リリース）は `talent-interview-profile-poc` と同趣旨とし、**マッチングアルゴリズムや人材との紐付けは本 spec では行わない**。

## Scope

- **In**: 案件の作成・一覧・単体取得・更新（最小フィールド集合）、入力検証と失敗時の利用者向け説明、既存 PoC との併存（ルーティング・パッケージ境界の明確化）、自動テストで主要ユースケースを検証可能にすること。
- **Out**: マッチング実行、ランキング、人材との関連付け、発注ワークフロー、権限モデル、課金、通知、複数テナント。

## Boundary Candidates

- **案件ドメイン**: 案件エンティティとそのライフサイクル（下書き／公開など必要最小限の状態があれば要件で定義）。
- **プレゼンテーション**: HTTP 契約と（採用する場合）Web 画面。
- **永続化**: 案件の保存と整合性（設計でテーブル／リポジトリに落とす）。

## Out of Boundary

- `talent-interview-profile-poc` が担う面談テキスト・YAML テンプレ・抽出・プロフィール反映の変更（バグ修正以外の機能拡張は本 spec の責務外）。
- クラウド LLM 前提の推論、音声認識。

## Upstream / Downstream

- **Upstream**: 既存モノレポのビルド／起動手順（`Makefile`、各 `README`）、既存 API のホスト・ポート慣習。
- **Downstream（将来）**: 人材と案件のマッチング、発注側ダッシュボード、検索・フィルタ高度化。別 spec またはフェーズ 2。

## Existing Spec Touchpoints

- **Adjacent**: `talent-interview-profile-poc` — 人材データと案件データは **本フェーズでは結合しない**。URL 名前空間やパッケージ名で衝突しないこと。
- **Extends**: なし（新規境界）。

## Constraints

- プロダクト steering（`product.md`）の「人材とプロジェクトをマッチング」の長期像に矛盾しない最小縦割りとする。
- 技術スタックは steering `tech.md` の実態（Python 3.14 / FastAPI / TS strict / SQLite）に整合させる（詳細は design フェーズ）。
