# Requirements Document: project-registration-poc

## Introduction

**context-matching** の利用者（運用者または将来の発注側ユーザー）が、マッチングの前提となる **案件**（募集内容・求める条件を説明する単位）をシステムに登録し、あとから一覧・詳細で参照・更新できるようにする。現状、リポジトリに実装されているのは人材と面談プロフィールに関する PoC のみであり、案件側のデータが存在しない。本仕様は **案件マスタの最小縦割り** を追加し、プロダクト steering が想定する「人材とプロジェクト（案件）のマッチング」の土台を用意する。技術選定や内部構造は設計書に委ね、本書は利用者・運用者が観察できる振る舞いと境界に限定する。

## Boundary Context

- **In scope**: 案件の作成、一覧、単体参照、更新、入力検証と失敗時の分かる説明、既存の人材面談 PoC と併存する利用導線（同一リポジトリ内での追加として利用者が両方を使い分けられること）。
- **Out of scope**: マッチング計算、人材との関連付け、案件への応募・選考ワークフロー、権限・ロールの本格モデル、課金、通知、全文検索エンジン、複数組織テナント分離。
- **Adjacent expectations**: `talent-interview-profile-poc` が提供する人材・面談・抽出・プロフィールの API と画面は **本リリースで変更しない**（案件機能の追加が既存の主要操作列を破壊しないこと）。案件と人材の **同一画面での突合せ** は含めない。

## Requirements

### Requirement 1: 案件の新規登録

**Objective:** As a 運用者, I want 案件を必須項目を満たして登録できる, so that 後続のマッチングや説明の土台となるデータがシステムに存在する。

#### Acceptance Criteria

1. When 運用者が登録に必要な最小項目をすべて満たして送信したとき、 the 案件登録 PoC shall 新しい案件を受け付け、システム内で一意に再参照できる識別子を返す。
2. If 必須項目のいずれかが欠ける、または許容されない形式であるとき、 the 案件登録 PoC shall 登録を拒否し、どの入力をどう直せばよいか追える情報を返す。
3. The 案件登録 PoC shall 登録時点の案件内容を、識別子を用いた参照で後から取得した内容と整合する形で保持する。

### Requirement 2: 案件の一覧と詳細参照

**Objective:** As a 運用者, I want 登録済み案件を一覧し個別を開ける, so that どの案件が存在するか把握し詳細を確認できる。

#### Acceptance Criteria

1. When 運用者が案件一覧を要求したとき、 the 案件登録 PoC shall 登録済み案件の識別子と一覧表示に必要な要約情報を返す。
2. When 運用者が既存の識別子で案件詳細を要求したとき、 the 案件登録 PoC shall その案件の保存内容を返す。
3. If 存在しない識別子で詳細が要求されたとき、 the 案件登録 PoC shall 存在しないことを判別できる形で失敗させる。

### Requirement 3: 案件の更新

**Objective:** As a 運用者, I want 既存案件の内容を更新できる, so that 募集条件の変更をデータに反映できる。

#### Acceptance Criteria

1. When 運用者が既存の識別子と有効な更新内容を送信したとき、 the 案件登録 PoC shall 当該案件の保存内容を更新し、更新後の内容を参照で取得できる。
2. If 更新内容が検証ルールを満たさないとき、 the 案件登録 PoC shall 更新を拒否し、修正に使える情報を返す。
3. If 存在しない識別子への更新が試みられたとき、 the 案件登録 PoC shall 存在しないことを判別できる形で失敗させる。

### Requirement 4: 最小 Web からの登録・参照・更新

**Objective:** As a 運用者, I want ブラウザから案件を登録・一覧・詳細・更新できる, so that API ツールなしで PoC を演示できる。

#### Acceptance Criteria

1. Where 最小 Web UI が提供される, When 運用者が画面の導線に従って案件を登録したとき, the 案件登録 PoC shall Requirement 1 の受入基準を満たす結果が画面から確認できる。
2. Where 最小 Web UI が提供される, When 運用者が一覧から案件を選択したとき, the 案件登録 PoC shall Requirement 2 に相当する情報が画面に表示される。
3. Where 最小 Web UI が提供される, When 運用者が編集画面から有効な変更を保存したとき, the 案件登録 PoC shall Requirement 3 に相当する結果が画面から確認できる。
4. If API がエラーを返したとき, the 案件登録 PoC shall 利用者向けのエラーメッセージを画面に示す。

### Requirement 5: 対象外機能の非提供

**Objective:** As a プロダクト責任者, I want 本 PoC の境界外であるマッチング等が誤って提供されない, so that スコープの誤解を防ぐ。

#### Acceptance Criteria

1. The 案件登録 PoC shall マッチング結果・ランキング・人材との自動紐付けを約束する利用者向け操作を提供しない。
2. If 利用者が発注ワークフローや課金など、本仕様の対象外に見える操作を求めた場合に備えた問い合わせがあっても, the 案件登録 PoC shall 本仕様書に記載された In/Out の範囲を超える振る舞いを追加しない（対応は将来 spec に委ねる）。
